# 56. Неблокирующий сетевой обмен. Селекторы

**Неблокирующий I/O** (Java NIO, пакет `java.nio.channels`) позволяет **одному потоку** обслуживать тысячи соединений: вместо блокировки на каждом сокете поток опрашивает **селектор**, который сообщает, какие каналы *готовы* к операциям.

## Зачем: проблема thread-per-connection

Классическая модель «поток на клиента» (`ServerSocket` + `accept()` + поток) упирается в ресурсы: каждый поток — это стек (по умолчанию ~1 МБ), переключения контекста, лимиты ОС. При 10 000+ соединений (**проблема C10K**) модель деградирует. NIO решает это **мультиплексированием**: один event loop обрабатывает все каналы.

## Триада NIO

* **`Channel`** — двунаправленный аналог потока (`SocketChannel`, `ServerSocketChannel`, `DatagramChannel`);
* **`Buffer`** (`ByteBuffer`) — контейнер данных, через который идёт чтение/запись;
* **`Selector`** — мультиплексор: отслеживает множество каналов и возвращает готовые.

Под капотом `Selector` опирается на системные вызовы ОС: **epoll** (Linux), **kqueue** (macOS/BSD), что и даёт масштабируемость.

## Регистрация: `SelectionKey`

Канал переводится в неблокирующий режим и регистрируется в селекторе с **маской интересующих операций**:

```java
channel.configureBlocking(false);          // обязательно ДО register
SelectionKey key = channel.register(selector, SelectionKey.OP_READ);
```

| Константа | Готовность | Кто использует |
|---|---|---|
| `OP_ACCEPT` | входящее подключение | `ServerSocketChannel` |
| `OP_CONNECT` | завершение `connect()` | клиентский `SocketChannel` |
| `OP_READ` | есть данные для чтения | `SocketChannel`, `DatagramChannel` |
| `OP_WRITE` | сокет готов принять данные | те же |

**Тонкость с `OP_WRITE`:** сокет почти всегда готов к записи, поэтому регистрировать `OP_WRITE` постоянно нельзя — селектор будет «жужжать» вхолостую. Его включают только когда буфер отправки заполнился (запись вернула 0), и снимают после успешной отправки.

В `key.attach(obj)` / `key.attachment()` удобно хранить **состояние клиента** (буферы, стадию протокола, имя пользователя) — это заменяет «состояние в стеке потока» из блокирующей модели.

## Скелет event loop

```java
Selector selector = Selector.open();
ServerSocketChannel server = ServerSocketChannel.open();
server.bind(new InetSocketAddress(8080));
server.configureBlocking(false);
server.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    selector.select();                            // блокируется до готовности
    Iterator<SelectionKey> it = selector.selectedKeys().iterator();
    while (it.hasNext()) {
        SelectionKey key = it.next();
        it.remove();                              // ОБЯЗАТЕЛЬНО: селектор сам не чистит
        if (key.isAcceptable()) {
            SocketChannel client = server.accept();
            client.configureBlocking(false);
            client.register(selector, SelectionKey.OP_READ);
        } else if (key.isReadable()) {
            // читаем из (SocketChannel) key.channel() в ByteBuffer
        }
    }
}
```

* `select()` блокируется до готовности хотя бы одного канала (есть `select(timeout)` и неблокирующий `selectNow()`);
* `selectedKeys()` возвращает **изменяемое** множество — обработанные ключи нужно удалять вручную, иначе событие «прилипнет»;
* `selector.wakeup()` — разбудить поток, спящий в `select()`, из другого потока (например, чтобы перерегистрировать интересы).

## Преимущества и цена

* один поток ↔ тысячи соединений, экономия памяти и переключений контекста;
* но: код превращается в **конечный автомат** (состояние нужно хранить явно), частичные чтения/записи нужно обрабатывать вручную. Поэтому в продакшене обычно используют надстройки — **Netty**, где event loop уже реализован.

### Шпаргалка

* NIO-мультиплексирование решает проблему C10K: один поток + `Selector` вместо потока на клиента.
* Триада: `Channel` + `ByteBuffer` + `Selector` (под капотом — epoll/kqueue).
* `configureBlocking(false)` обязательно **до** `register()`; интересы — `OP_ACCEPT/CONNECT/READ/WRITE`.
* `OP_WRITE` регистрируют только при невозможности записать, иначе busy loop.
* В цикле: `select()` → `selectedKeys()` → обработка → **`iterator.remove()`**; состояние клиента — в `attachment`.
