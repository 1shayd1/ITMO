# 58. Протокол UDP. Класс DatagramChannel

`DatagramChannel` (пакет `java.nio.channels`) — **NIO-канал для UDP**. Работает с датаграммами через `ByteBuffer`, поддерживает **блокирующий и неблокирующий** режимы и интегрируется с `Selector` — в отличие от классического `DatagramSocket`.

## Напоминание об UDP

* каждое сообщение — отдельная **датаграмма** с сохранением границ;
* нет соединения, нет гарантий доставки, порядка и отсутствия дубликатов;
* теоретический максимум полезной нагрузки ~**65 507 байт**, но на практике датаграммы держат меньше **MTU** (~1472 байта для Ethernet), чтобы избежать IP-фрагментации и роста потерь.

## Создание и привязка

```java
DatagramChannel ch = DatagramChannel.open();
ch.bind(new InetSocketAddress(9999));   // для приёма; отправителю можно не бинди́ться
```

## Обмен без соединения: `send` / `receive`

```java
// Отправка
ByteBuffer buf = ByteBuffer.wrap("ping".getBytes(StandardCharsets.UTF_8));
ch.send(buf, new InetSocketAddress("localhost", 9999));

// Приём
ByteBuffer in = ByteBuffer.allocate(1500);
SocketAddress sender = ch.receive(in);   // вернёт адрес отправителя
in.flip();                                // подготовить буфер к чтению
```

* `receive()` кладёт в буфер **одну датаграмму**; если буфер меньше датаграммы, **лишние байты молча отбрасываются** — буфер должен быть достаточного размера;
* возвращаемый `SocketAddress` позволяет ответить отправителю;
* в неблокирующем режиме `receive()` возвращает `null`, если датаграмм нет.

Не забывать о цикле жизни `ByteBuffer`: `flip()` перед чтением из буфера, `clear()` перед повторным приёмом.

## «Подключённый» режим: `connect`

```java
ch.connect(new InetSocketAddress("localhost", 9999));
ch.write(buf);   // только этому адресату
ch.read(in);     // пакеты от других отправителей отбрасываются ОС
```

`connect()` для UDP **не устанавливает соединение** (handshake нет) — он лишь фиксирует единственного пира: появляются `read()`/`write()`, фильтрация чужих пакетов и небольшой выигрыш в производительности (проверка адреса один раз). `disconnect()` снимает фиксацию.

## Неблокирующий режим и `Selector`

Главное преимущество перед `DatagramSocket`:

```java
ch.configureBlocking(false);
ch.register(selector, SelectionKey.OP_READ);
```

Один поток может мультиплексировать множество UDP- и TCP-каналов в общем event loop.

## `DatagramChannel` vs `DatagramSocket`

| Свойство | `DatagramSocket` | `DatagramChannel` |
|---|---|---|
| Пакет | `java.net` (IO) | `java.nio.channels` (NIO) |
| Данные | `DatagramPacket` (массив байт) | `ByteBuffer` |
| Неблокирующий режим | Нет | Да |
| Работа с `Selector` | Нет | Да |
| Подключённый режим | `connect()` (фильтрация) | `connect()` + `read`/`write` |

### Шпаргалка

* `DatagramChannel` — UDP из NIO: `ByteBuffer`, блокирующий/неблокирующий режимы, `Selector`.
* `send(buf, addr)` / `receive(buf)`; `receive` возвращает адрес отправителя, лишнее в малом буфере **отбрасывается**.
* `connect()` фиксирует пира (без handshake) → доступны `read()`/`write()`, чужие пакеты фильтруются.
* Датаграмму держат меньше MTU (~1472 байта), теоретический предел ~65 507.
* Помнить про `flip()`/`clear()` у `ByteBuffer`.
