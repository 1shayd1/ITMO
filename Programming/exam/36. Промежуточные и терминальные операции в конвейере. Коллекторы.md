# 36. Промежуточные и терминальные операции в конвейере. Коллекторы

В `Stream API` все операции делятся на **промежуточные** (intermediate) и **терминальные** (terminal). Это разделение определяет **ленивость** и **структуру конвейера**.

---

## 🔄 Ключевое различие

| Характеристика | Промежуточные | Терминальные |
|----------------|---------------|---------------|
| Возвращают | Новый `Stream` | Результат (`List`, `Integer`, `void` и т.д.) |
| Ленивость | ✅ (не выполняются сразу) | ❌ (запускают весь конвейер) |
| Количество в конвейере | Много | Один (и только один) |
| Без терминала | Бесполезны (ничего не произойдёт) | Невозможно создать конвейер |

```java
// ❌ ничего не выполнится (нет терминала)
list.stream().filter(x -> x > 5).map(x -> x * 2);

// ✅ терминал запускает всю цепочку
list.stream().filter(x -> x > 5).map(x -> x * 2).toList();
```

## 🔧 Промежуточные операции (подробно)

### Stateless (без сохранения состояния) — эффективны для параллельных потоков

| Операция | Что делает | Пример |
|----------|------------|--------|
| `filter(Predicate)` | Оставляет только элементы, подходящие под условие | `.filter(s -> s.length() > 3)` |
| `map(Function)` | Преобразует каждый элемент | `.map(String::toUpperCase)` |
| `flatMap(Function)` | Разворачивает вложенные коллекции в плоский поток | `.flatMap(list -> list.stream())` |
| `peek(Consumer)` | Побочное действие (отладка) | `.peek(System.out::println)` |

### Stateful (запоминают состояние) — требуют памяти

| Операция | Что делает | Особенность | Complexity |
|----------|------------|-------------|------------|
| `distinct()` | Убирает дубликаты | Использует `HashSet` под капотом | O(n) по памяти |
| `sorted()` | Сортирует по натуральному порядку | Загружает все элементы в массив | O(n log n) time, O(n) memory |
| `sorted(Comparator)` | Сортировка с компаратором | Тоже требует полной загрузки | O(n log n) time, O(n) memory |
| `limit(long n)` | Оставляет первые n элементов | Short-circuiting | O(1) памяти при последовательном |
| `skip(long n)` | Пропускает первые n элементов | Требует счётчика | O(n) времени |

### Примеры stateful операций с пояснением

```java
// distinct — хранит все просмотренные элементы в HashSet
stream.distinct()
      .forEach(...); // память растёт с каждым новым уникальным элементом

// sorted — загружает все элементы в массив, сортирует, затем итерирует
stream.sorted()
      .limit(5)      // ❌ limit не спасёт — sorted всё равно загрузит всё
      .toList();

// limit + sorted в правильном порядке
stream.limit(100)    // ✅ сначала ограничиваем
      .sorted()      // потом сортируем только 100 элементов
      .toList();
```

## 🚀 Терминальные операции (полная классификация)

### 1. Агрегирующие (редуцирующие)

| Операция | Возвращает | Пример | Complexity |
|----------|------------|--------|-------------|
| `count()` | `long` | `.count()` | O(n) |
| `min(Comparator)` / `max()` | `Optional<T>` | `.min(String::compareTo)` | O(n) |
| `sum()` (IntStream) | `int` | `.sum()` | O(n) |

### 2. Collecting (сбор в коллекции)

| Операция | Возвращает | Пример |
|----------|------------|--------|
| `toList()` | `List<T>` | `.toList()` (Java 16+) |
| `collect(Collector)` | Зависит от коллектора | `.collect(Collectors.toSet())` |

### 3. Reducing (обобщённое свертывание)

```java
// reduce(T identity, BinaryOperator<T>)
int sum = list.stream().reduce(0, (a, b) -> a + b);

// reduce с Optional (без identity)
Optional<Integer> product = list.stream().reduce((a, b) -> a * b);
```

### 4. Проверяющие (matching / finding)

| Операция | Что делает | Short-circuiting |
|----------|------------|------------------|
| `anyMatch(Predicate)` | Хотя бы один подходит | ✅ (останавливается при первом true) |
| `allMatch(Predicate)` | Все подходят | ✅ (останавливается при первом false) |
| `noneMatch(Predicate)` | Ни один не подходит | ✅ (останавливается при первом true) |
| `findFirst()` | Первый элемент (с учётом порядка) | ✅ |
| `findAny()` | Любой элемент (быстрее для parallel) | ✅ |

### 5. Действия (side-effect)

| Операция | Возвращает | Примечание |
|----------|------------|-------------|
| `forEach(Consumer)` | `void` | Не гарантирует порядок в parallel |
| `forEachOrdered(Consumer)` | `void` | Гарантирует порядок (медленнее) |
| `toArray()` | `T[]` | Преобразование в массив |

### Примеры терминальных операций

```java
// count
long cnt = list.stream().filter(x -> x > 0).count();

// anyMatch (short-circuit)
boolean hasNegative = list.stream().anyMatch(x -> x < 0);

// findFirst с Optional
String first = list.stream().filter(s -> s.length() > 5).findFirst().orElse("none");

// toArray (с генератором для типизированного массива)
String[] arr = list.stream().toArray(String[]::new);
```

## 📦 Коллекторы (Collectors) — подробно

`Collectors` — это фабрика реализаций интерфейса `Collector<T, A, R>`, где:
- `T` — тип элементов потока
- `A` — аккумулятор (промежуточное состояние, mutable)
- `R` — конечный результат

### Основные коллекторы

| Коллектор | Возвращает | Пример | Complexity |
|-----------|------------|--------|-------------|
| `toList()` | `List<T>` | `.collect(Collectors.toList())` | O(n) |
| `toSet()` | `Set<T>` | `.collect(Collectors.toSet())` | O(n), без дубликатов |
| `toMap(keyMapper, valueMapper)` | `Map<K,V>` | `.collect(Collectors.toMap(Function.identity(), String::length))` | O(n), конфликт ключей → исключение |
| `toMap(keyMapper, valueMapper, mergeFunction)` | `Map<K,V>` | `.collect(Collectors.toMap(Function.identity(), v->1, Integer::sum))` | Разрешает коллизии |
| `joining()` | `String` | `.collect(Collectors.joining(", "))` | O(n) |
| `joining(delimiter, prefix, suffix)` | `String` | `.collect(Collectors.joining(", ", "[", "]"))` | → `[a, b, c]` |
| `counting()` | `Long` | `.collect(Collectors.counting())` | Альтернатива `.count()` |
| `summingInt(ToIntFunction)` | `Integer` | `.collect(Collectors.summingInt(String::length))` | O(n) |
| `averagingInt()` | `Double` | `.collect(Collectors.averagingInt(String::length))` | O(n) |
| `groupingBy(Function)` | `Map<K, List<T>>` | `.collect(Collectors.groupingBy(String::length))` | O(n), мапа списков |
| `groupingBy(Function, downstream)` | `Map<K, D>` | `.collect(Collectors.groupingBy(String::length, Collectors.counting()))` | Группировка + агрегация |
| `partitioningBy(Predicate)` | `Map<Boolean, List<T>>` | `.collect(Collectors.partitioningBy(s -> s.length() > 3))` | Разбивает на 2 части |
| `toCollection(Supplier)` | `Collection<T>` | `.collect(Collectors.toCollection(LinkedHashSet::new))` | Контроль над типом коллекции |

### Примеры коллекторов в действии

```java
import java.util.*;
import java.util.stream.Collectors;

// toList — Java 16+ упрощённый
List<String> list = stream.toList();           // проще, чем .collect(toList())

// toMap с разрешением конфликтов
Map<String, Integer> map = list.stream()
    .collect(Collectors.toMap(
        s -> s,                     // ключ
        String::length,             // значение
        (v1, v2) -> v1              // если ключ дублируется, оставляем первый
    ));

// groupingBy с downstream-коллектором
Map<Integer, Long> lengthCount = list.stream()
    .collect(Collectors.groupingBy(
        String::length,             // ключ = длина строки
        Collectors.counting()       // значение = количество строк этой длины
    ));
// {3=2, 4=1, 5=3}

// partitioningBy — всегда 2 ключа: true/false
Map<Boolean, List<String>> partitioned = list.stream()
    .collect(Collectors.partitioningBy(s -> s.length() > 4));

// multi-level groupingBy
Map<Integer, Map<Character, List<String>>> deep = list.stream()
    .collect(Collectors.groupingBy(
        String::length,
        Collectors.groupingBy(s -> s.charAt(0))
    ));

// joining с форматированием
String result = list.stream()
    .collect(Collectors.joining(", ", "Prefix: ", ":Suffix"));
// Prefix: a, b, c:Suffix
```

## ⚙️ Подкапотные детали

### Как работает `collect()` внутри?

1. `collect(Collector)` вызывает у коллектора:
   - `supplier()` — создаёт новый аккумулятор (например, `ArrayList::new`)
   - `accumulator()` — добавляет элемент в аккумулятор
   - `combiner()` — объединяет два аккумулятора (для параллельных потоков)
   - `finisher()` — преобразует аккумулятор в результат

```java
// Пример ручной реализации Collector (упрощённо)
Collector<String, List<String>, List<String>> toListCollector = 
    Collector.of(
        ArrayList::new,           // supplier
        List::add,                // accumulator
        (left, right) -> { left.addAll(right); return left; }, // combiner
        Function.identity()       // finisher (сохраняем аккумулятор)
    );
```

### Ленивость и оптимизации

- `sorted()` + `limit(10)` → оптимизируется в `nlargest` (heap-based), а не полная сортировка O(n log n) → O(n log 10)
- `map().filter().map()` → некоторые JVM-оптимизации схлопывают цепочки
- `parallelStream()` использует `ForkJoinPool.commonPool()` (по умолчанию)

## ⚠️ Частые ошибки и контракты

### 1. Нельзя модифицировать источник во время обработки

```java
List<String> list = new ArrayList<>(Arrays.asList("a", "b"));
list.stream().peek(s -> list.add("c")).toList(); // ❌ ConcurrentModificationException
```

### 2. `parallelStream()` не всегда быстрее

- Маленькие коллекции → overhead перевешивает
- Неассоциативные операции (например, `reduce` для вычитания) → неверный результат
- Stateful операции (`sorted`, `distinct`) → узкое место

### 3. В `collect(toMap())` дубликаты ключей → исключение

```java
// ❌ IllegalStateException (дубликат ключа "a")
Stream.of("a", "a").collect(Collectors.toMap(s -> s, s -> 1));

// ✅ разрешаем конфликт
Stream.of("a", "a").collect(Collectors.toMap(s -> s, s -> 1, (old, new) -> old + new));
// {a=2}
```

### 4. После терминала поток нельзя переиспользовать

```java
Stream<String> s = list.stream();
s.count();
s.toList(); // ❌ IllegalStateException
```

## 🧠 Полный пример: анализ логов

```java
record LogEntry(String level, String message, int length) {}

List<LogEntry> logs = List.of(
    new LogEntry("ERROR", "Null pointer", 12),
    new LogEntry("INFO", "Started", 7),
    new LogEntry("ERROR", "IO failed", 9),
    new LogEntry("DEBUG", "Config loaded", 12)
);

// Группировка по уровню, суммарная длина сообщений
Map<String, Integer> sumByLevel = logs.stream()
    .collect(Collectors.groupingBy(
        LogEntry::level,
        Collectors.summingInt(LogEntry::length)
    ));
// {ERROR=21, INFO=7, DEBUG=12}

// Топ-3 самых длинных сообщения
List<String> topMessages = logs.stream()
    .sorted((a, b) -> Integer.compare(b.length(), a.length()))
    .limit(3)
    .map(LogEntry::message)
    .toList();

// Проверка: есть ли хоть одно ERROR-сообщение длиннее 10 символов
boolean hasLongError = logs.stream()
    .filter(e -> e.level().equals("ERROR"))
    .anyMatch(e -> e.length() > 10);
```

### Шпаргалка

* **Промежуточные операции ленивы** — без терминала ничего не работает.
* **Терминал только один** — запускает весь конвейер и закрывает поток.
* **Stateful операции** (`sorted`, `distinct`) запоминают все элементы → O(n) памяти.
* **Коллекторы** — мощные терминалы для группировки, маппинга и свёртки.
* **`groupingBy` + downstream** — создаёт иерархические структуры одним вызовом.
* **`parallelStream()` ≠ магия** — работает только для stateless, ассоциативных операций.
* **Дубликаты ключей в `toMap()`** → `IllegalStateException` без merge-функции.
