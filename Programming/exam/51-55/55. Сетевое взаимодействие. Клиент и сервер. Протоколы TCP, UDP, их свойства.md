# 55. Сетевое взаимодействие. Клиент и сервер. Протоколы TCP, UDP, их свойства

Сетевое программирование в Java строится на модели **клиент–сервер** поверх протоколов **транспортного уровня** (модель TCP/IP): `TCP` и `UDP`. Оба работают поверх IP и адресуют приложения через **порты** (0–65535); пара *(IP-адрес, порт)* однозначно определяет конечную точку.

## Клиент и сервер

* **Сервер** пассивно **слушает** заранее известный порт и принимает входящие подключения/пакеты.
* **Клиент** активно **инициирует** обмен: подключается к адресу и порту сервера (его собственный порт обычно выдаётся ОС — эфемерный).

## `TCP` (Transmission Control Protocol)

Протокол **с установлением соединения**, передающий **непрерывный поток байтов**.

### Как обеспечивается надёжность

* **Three-way handshake** при установке соединения: `SYN` → `SYN-ACK` → `ACK`;
* каждый байт нумеруется (**sequence numbers**); получатель подтверждает приём (**ACK**), потерянные сегменты **ретранслируются**;
* порядок восстанавливается по номерам; дубликаты отбрасываются; целостность проверяется контрольной суммой;
* **flow control** (скользящее окно — не перегружать получателя) и **congestion control** (не перегружать сеть);
* заголовок — минимум **20 байт**.

### Важное следствие: поток без границ

`TCP` не сохраняет границы сообщений: три вызова `write()` могут прийти как один блок и наоборот. Прикладной протокол должен сам вводить **framing** — длину-префикс, разделители и т.п.

## `UDP` (User Datagram Protocol)

Протокол **без установления соединения**, передающий **отдельные датаграммы**.

### Свойства

* нет гарантии доставки, порядка и отсутствия дубликатов;
* каждая датаграмма — самостоятельное сообщение с сохранением границ;
* минимальные накладные расходы: заголовок **8 байт**, нет handshake — ниже задержка;
* поддерживает **broadcast** и **multicast** (TCP — только точка-точка).

## Сравнение

| Свойство                   | `TCP`                     | `UDP`                           |
| -------------------------- | ------------------------- | ------------------------------- |
| Соединение                 | Да (handshake)            | Нет                             |
| Надёжность доставки        | Да (ACK + ретрансмиссия)  | Нет                             |
| Порядок                    | Гарантируется             | Нет                             |
| Модель данных              | Поток байтов              | Датаграммы с границами          |
| Контроль потока/перегрузки | Да                        | Нет                             |
| Заголовок                  | 20+ байт                  | 8 байт                          |
| Broadcast/multicast        | Нет                       | Да                              |
| Типичные применения        | HTTP(S), почта, файлы, БД | DNS, VoIP, стриминг, игры, QUIC |

Современный пример: протокол **QUIC** (основа HTTP/3) реализует надёжность *поверх* UDP, забирая лучшее из обоих миров.

## Маппинг на Java API

| Протокол | Классический I/O | NIO |
|---|---|---|
| `TCP` | `Socket`, `ServerSocket` | `SocketChannel`, `ServerSocketChannel` |
| `UDP` | `DatagramSocket`, `DatagramPacket` | `DatagramChannel` |

### Шпаргалка

* Сервер слушает известный порт, клиент инициирует обмен; конечная точка = IP + порт.
* `TCP`: соединение (3-way handshake), надёжность, порядок, **поток байтов без границ** → нужен framing.
* `UDP`: без соединения, без гарантий, **датаграммы с границами**, заголовок 8 байт, broadcast/multicast.
* Выбор: надёжность и целостность → TCP; минимальная задержка и допустимые потери → UDP.
* В Java: `Socket`/`ServerSocket` и `DatagramSocket` (IO), `SocketChannel`/`DatagramChannel` (NIO).
