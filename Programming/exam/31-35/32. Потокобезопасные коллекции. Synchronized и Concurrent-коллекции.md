# 32. Потокобезопасные коллекции. Synchronized и Concurrent-коллекции

В многопоточности обычные коллекции могут ломаться при одновременном доступе. Для этого существуют потокобезопасные варианты.

## Почему обычные коллекции небезопасны

`ArrayList`, `HashMap` и другие коллекции из `java.util` не синхронизированы. При одновременном доступе из нескольких потоков возможны:

- **Race condition** — два потока одновременно модифицируют структуру, результат непредсказуем.
- **ConcurrentModificationException** — один поток итерируется, другой изменяет коллекцию.
- **Потеря данных** — операции типа `put` или `add` не атомарны, часть изменений может быть потеряна.

```java
// Пример небезопасного кода:
List<Integer> list = new ArrayList<>();

Runnable task = () -> {
    for (int i = 0; i < 1000; i++) {
        list.add(i); // Race condition! Разные потоки могут затереть друг друга
    }
};

Thread t1 = new Thread(task);
Thread t2 = new Thread(task);
t1.start(); t2.start();
// Результат непредсказуем — ConcurrentModificationException или потеря элементов
```

---

## Два подхода

### Synchronized-обёртки

Это коллекции, обёрнутые в синхронизацию:

- `Collections.synchronizedList(...)`
- `Collections.synchronizedSet(...)`
- `Collections.synchronizedMap(...)`

Они защищают коллекцию через единый монитор.

**Как работают:** каждый метод оборачивается в `synchronized(mutex)`, где `mutex` — внутренний объект-замок. Все операции выполняются строго по одной.

```java
List<String> syncList = Collections.synchronizedList(new ArrayList<>());

syncList.add("hello");   // потокобезопасно
syncList.get(0);         // потокобезопасно

// ВАЖНО: итерация требует ручной синхронизации!
synchronized (syncList) {
    for (String s : syncList) {
        System.out.println(s);
    }
}
```

> **Почему итерация не защищена автоматически?** Потому что итератор — отдельный объект, и между вызовами `hasNext()` и `next()` другой поток может изменить коллекцию. Поэтому весь блок итерации нужно синхронизировать вручную.

---

### Concurrent-коллекции

Это специальные коллекции из `java.util.concurrent`, рассчитанные на одновременный доступ:

- `ConcurrentHashMap`
- `CopyOnWriteArrayList`
- `CopyOnWriteArraySet`
- `ConcurrentLinkedQueue`
- `ConcurrentLinkedDeque`
- `LinkedBlockingQueue`
- `ArrayBlockingQueue`

Они используют более тонкие механизмы синхронизации (сегментные блокировки, CAS-операции, copy-on-write), что даёт лучшую производительность при высокой конкуренции.

---

## Сравнение

|Тип|Идея|Производительность|
|---|---|---|
|Synchronized-обёртки|Грубая синхронизация|Ниже при конкуренции|
|Concurrent-коллекции|Более тонкая конкурентность|Обычно выше|

## `ConcurrentHashMap`

Одна из самых важных concurrent-коллекций. Позволяет многим потокам безопасно читать и писать данные с меньшими блокировками, чем у синхронизированной `HashMap`.

### Как устроена

- В Java 8+ использует **CAS (Compare-And-Swap)** и синхронизацию на уровне отдельных бакетов (а не всей таблицы).
- Операции чтения (`get`) вообще не блокируются.
- Операции записи блокируют только конкретный бакет.

```java
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

map.put("a", 1);
map.put("b", 2);

// Атомарные составные операции:
map.putIfAbsent("c", 3);          // добавит только если ключа нет
map.computeIfAbsent("d", k -> k.length()); // вычислит значение если ключа нет
map.merge("a", 10, Integer::sum); // атомарно обновит значение

// Безопасная итерация (не бросает ConcurrentModificationException):
map.forEach((k, v) -> System.out.println(k + " = " + v));
```

### Чего нельзя делать

```java
// НЕ атомарно! Между get и put другой поток может вмешаться:
if (!map.containsKey("key")) {
    map.put("key", value); // ОПАСНО
}

// Правильно:
map.putIfAbsent("key", value); // атомарная операция
```

### `null` не допускается

```java
map.put(null, 1);    // NullPointerException!
map.put("a", null);  // NullPointerException!
```

## `CopyOnWriteArrayList`

Подходит, когда чтений много, а изменений мало. При модификации создаётся новая копия массива.

### Как устроена

- При каждом `add`, `set`, `remove` создаётся **полная копия** внутреннего массива.
- Чтения (`get`, итерация) всегда работают со старым снапшотом — блокировок нет вообще.
- Изменения защищены `ReentrantLock`.

```java
CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>();

list.add("a");
list.add("b");

// Итерация безопасна даже без синхронизации:
for (String s : list) {
    System.out.println(s);
    list.add("c"); // Не вызовет ConcurrentModificationException!
    // Но "c" не появится в текущей итерации — мы идём по старому снапшоту
}
```

### Недостатки

- **Дорогие записи** — каждое изменение копирует весь массив. O(n) памяти и времени.
- **Слабая консистентность** — читатель видит снапшот на момент начала итерации, не самые свежие данные.

### Когда подходит

- Списки слушателей (event listeners).
- Кэши, которые редко обновляются.
- Read-heavy сценарии с редкими записями.

## `ConcurrentLinkedQueue`

Неблокирующая потокобезопасная очередь на основе алгоритма **Michael-Scott** с CAS-операциями.

```java
ConcurrentLinkedQueue<String> queue = new ConcurrentLinkedQueue<>();

queue.offer("task1"); // добавить в хвост
queue.offer("task2");

String task = queue.poll(); // забрать из головы (или null если пусто)
String head = queue.peek(); // посмотреть голову без удаления
```

- Не блокирует потоки — вместо этого повторяет CAS до успеха.
- `size()` работает за O(n) — не используй в hot path.
- Подходит для producer-consumer без ограничения размера.

## Blocking-коллекции

Отдельная категория — **блокирующие очереди** (`BlockingQueue`), которые умеют ждать:

```java
BlockingQueue<String> queue = new LinkedBlockingQueue<>(100); // ёмкость 100

// Producer: заблокируется если очередь полна
queue.put("task");

// Consumer: заблокируется если очередь пуста
String task = queue.take();

// Без блокировки (вернёт false/null если нельзя):
boolean added = queue.offer("task");
String t = queue.poll();
```

|Класс|Особенность|
|---|---|
|`LinkedBlockingQueue`|Связный список, опционально ограниченный|
|`ArrayBlockingQueue`|Массив фиксированной ёмкости|
|`PriorityBlockingQueue`|Очередь с приоритетом|
|`SynchronousQueue`|Передача напрямую от producer к consumer|
|`DelayQueue`|Элементы доступны только после задержки|

---

## Когда что использовать

- Для простых случаев — synchronized-обёртки.
- Для высокой конкуренции — concurrent-коллекции.
- Для частых чтений и редких изменений — `CopyOnWriteArrayList`.
- Для producer-consumer паттерна — `BlockingQueue`.
- Для конкурентной мапы — `ConcurrentHashMap`.

---

## Шпаргалка

- Обычные коллекции не потокобезопасны.
- `Collections.synchronized...` даёт простую защиту, но итерацию нужно синхронизировать вручную.
- `ConcurrentHashMap` — основной выбор для конкурентных мап; не допускает `null`.
- `CopyOnWriteArrayList` хорош при редких изменениях — дорогие записи, дешёвые чтения.
- `ConcurrentLinkedQueue` — неблокирующая очередь на CAS.
- `BlockingQueue` — для паттерна producer-consumer с ожиданием.
- Concurrent-коллекции обычно масштабируются лучше синхронизированных обёрток.
- Атомарные составные операции (`putIfAbsent`, `computeIfAbsent`) предпочтительнее ручных `if + put`.
