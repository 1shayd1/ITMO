# 53. Система модулей в Java. Провайдеры служб

**JPMS** (Java Platform Module System, проект Jigsaw, Java 9) — система модулей поверх пакетов и JAR'ов.

## Цели

* **сильная инкапсуляция**: `public` больше не значит «доступен всем» — снаружи видны только **экспортированные** пакеты;
* **надёжная конфигурация**: зависимости проверяются при запуске (отсутствующий модуль — ошибка старта, а не `ClassNotFoundException` посреди работы);
* **масштабируемая платформа**: сам JDK разбит на модули (`java.base`, `java.sql`...), `jlink` собирает урезанный рантайм только из нужных.

## Модуль и `module-info.java`

Модуль — именованная единица: набор пакетов + явные зависимости + экспорты. Дескриптор лежит в корне исходников:

```java
module com.example.app {
    requires java.sql;                  // зависимость
    requires transitive com.example.api; // зависимость, видимая и нашим клиентам
    requires static lombok;             // только на этапе компиляции

    exports com.example.app.api;        // пакет доступен всем
    exports com.example.app.spi to com.example.plugin; // адресный экспорт

    opens com.example.app.model;        // открыт для глубокой РЕФЛЕКСИИ в рантайме

    uses com.example.app.spi.Storage;                       // потребляем сервис
    provides com.example.app.spi.Storage
        with com.example.app.db.DbStorage;                  // поставляем реализацию
}
```

Ключевая разница `exports` vs `opens`: `exports` — доступ к public-типам на этапе компиляции и в рантайме; `opens` — разрешение **deep reflection** (в т.ч. `setAccessible` к private) в рантайме — нужно Hibernate/Jackson/Spring. `open module` открывает все пакеты.

## Виды модулей

* **именованные** — с `module-info.class`;
* **автоматические** — обычный JAR на module path: имя берётся из манифеста/имени файла, экспортирует всё (мост для миграции);
* **безымянный (unnamed)** — всё, что лежит на классическом classpath; читает все модули.

`java.base` требуется всеми неявно.

## Провайдеры служб (`ServiceLoader`)

Сервисный механизм разрывает связь «интерфейс ↔ реализация»: модуль-потребитель знает только интерфейс, реализации подключаются декларативно.

* Модуль API: интерфейс `Storage`;
* модуль-потребитель: `uses Storage;`
* модуль-провайдер: `provides Storage with DbStorage;` (реализация должна иметь public-конструктор без аргументов **или** статический метод `provider()`).

```java
ServiceLoader<Storage> loader = ServiceLoader.load(Storage.class);
for (Storage s : loader) {     // провайдеры инстанцируются ЛЕНИВО при итерации
    s.save(data);
}
Optional<Storage> first = loader.findFirst();
```

До JPMS тот же механизм работал через файлы `META-INF/services/полное.имя.Интерфейса` (так JDBC находит драйверы) — он действует и сейчас для classpath.

### Шпаргалка

* Модуль = пакеты + явные `requires`/`exports` в `module-info.java`; видимы снаружи только экспортированные пакеты.
* `requires transitive` — реэкспорт зависимости; `requires static` — только compile-time.
* `exports` ≠ `opens`: `opens` разрешает глубокую рефлексию (нужно Spring/Hibernate/Jackson).
* Сервисы: потребитель `uses X`, провайдер `provides X with Impl`; загрузка — `ServiceLoader.load(X.class)`, лениво.
* Виды модулей: именованные, автоматические (JAR на module path), unnamed (classpath).
