# 52. Безопасное хранение паролей. Предотвращение SQL-инъекций

Две классические уязвимости серверных приложений: утечка паролей из БД и внедрение SQL через пользовательский ввод.

## Безопасное хранение паролей

**Модель угрозы**: злоумышленник получил дамп БД и перебирает пароли офлайн. Хранение должно делать перебор максимально дорогим.

### Принципы

* **Никогда не хранить пароль в открытом виде** (и не логировать!) — только необратимый **хеш**;
* **Быстрые хеши не годятся**: `MD5`, `SHA-1`, `SHA-256` спроектированы быстрыми → GPU перебирает миллиарды кандидатов в секунду; против них также работают радужные таблицы;
* **Соль (salt)** — случайные байты, уникальные для каждого пользователя, хранятся рядом с хешем: `hash = H(salt + password)`. Соль ломает радужные таблицы и скрывает одинаковые пароли разных пользователей;
* **Медленные/«дорогие» KDF** — специальные алгоритмы для паролей с настраиваемой стоимостью:
  * `PBKDF2` (многократные итерации HMAC, есть в JDK: `SecretKeyFactory` `PBKDF2WithHmacSHA256`);
  * `bcrypt` (cost-фактор);
  * `scrypt`, **`Argon2`** (победитель Password Hashing Competition; *memory-hard* — дорог и по памяти, что бьёт по GPU/ASIC).
* Результат обычно хранится строкой формата `$algo$params$salt$hash` — параметры можно усиливать со временем.

### Java-специфика

* Пароль держать в **`char[]`, а не в `String`**: строки иммутабельны и живут в куче до GC (могут попасть в дамп памяти), массив можно затереть `Arrays.fill(pwd, '\0')` сразу после использования;
* сравнение хешей — **за постоянное время** (`MessageDigest.isEqual`), а не `equals` (защита от timing-атак);
* передача пароля по сети — только поверх TLS.

## SQL-инъекции

Инъекция возникает, когда ввод пользователя **конкатенируется** в SQL-строку и начинает интерпретироваться как код:

```java
// ОПАСНО: name = "' OR '1'='1" превращает запрос в SELECT * FROM users
String sql = "SELECT * FROM users WHERE name = '" + name + "'";
```

### Защита: `PreparedStatement`

```java
PreparedStatement ps = conn.prepareStatement(
    "SELECT * FROM users WHERE name = ?");
ps.setString(1, name);
ResultSet rs = ps.executeQuery();
```

**Почему это работает**: шаблон запроса и параметры передаются в СУБД **раздельно** — запрос разбирается/компилируется один раз, а значения подставляются как **типизированные данные**, которые в принципе не парсятся как SQL. Бонус — кэширование плана запроса (быстрее при повторных вызовах).

**Ограничение**: параметрами могут быть только **значения**. Имена таблиц/столбцов, `ORDER BY` параметризовать нельзя — для них допустим только **белый список** допустимых значений.

### Эшелонированная защита

* учётная запись БД с **минимальными правами** (только нужные таблицы/операции);
* ORM / именованные параметры (внутри всё равно prepared statements);
* валидация ввода по формату — полезна, но не замена параметризации;
* не показывать пользователю сырые SQL-ошибки.

### Шпаргалка

* Пароли: только хеш + **уникальная соль**; алгоритмы — медленные KDF (`bcrypt`, `PBKDF2`, `scrypt`, `Argon2`), а не быстрые MD5/SHA.
* Соль ломает радужные таблицы и маскирует совпадающие пароли; хранится рядом с хешем.
* В Java пароль — `char[]` с затиранием; сравнение хешей — constant-time.
* SQL-инъекция = конкатенация ввода в запрос; `PreparedStatement` шлёт код и данные раздельно → ввод не исполняется.
* Идентификаторы (таблицы/колонки) параметром не передать — только whitelist; плюс минимальные права у учётки БД.
