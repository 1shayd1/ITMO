# 35. Конвейеры и Stream API. Источники конвейера. Особенности обработки данных

`Stream API` — это декларативный способ обработки последовательностей данных через **конвейер операций**, основанный на ленивых вычислениях и функциональных преобразованиях. Поток не хранит данные, а описывает цепочку преобразований.

---

## 🔁 Модель конвейера (три этапа)

1. **Источник** (source) — откуда берутся данные  
2. **Промежуточные операции** (intermediate, `lazy`) — преобразования  
3. **Терминальная операция** (terminal, `eager`) — запускает выполнение и возвращает результат

---

## 📦 Источники конвейера (полный список)

| Источник | Способ получения |
|----------|------------------|
| Коллекция | `collection.stream()` / `parallelStream()` |
| Массив | `Arrays.stream(array)` |
| Файл | `Files.lines(path)` |
| Генератор | `Stream.generate()`, `Stream.iterate()` |
| Строка | `"abc".chars()` |
| Диапазон | `IntStream.range(1, 10)` |
| Значения | `Stream.of("a", "b")` |
| Регулярка | `Pattern.compile("\\s+").splitAsStream("a b c")` |
| Битовый сдвиг | `BitSet.stream()` |

### Примеры источников с кодом

```java
// коллекция
list.stream();

// массив
Arrays.stream(new int[]{1, 2, 3});

// файл — построчно (автозакрытие)
try (Stream<String> lines = Files.lines(Paths.get("file.txt"))) {
    lines.filter(l -> !l.isEmpty()).forEach(System.out::println);
}

// бесконечный генератор с limit
Stream.generate(Math::random).limit(5);

// итерация с условием (Java 9+)
Stream.iterate(1, n -> n < 100, n -> n * 2);
```

## ⚙️ Особенности обработки (под капотом)

### 1. Не изменяет исходную коллекцию (non-interference)
Поток работает с данными, но не модифицирует источник. Исключение — `peek()` с побочными эффектами, но это антипаттерн.

### 2. Ленивость (lazy evaluation)
Промежуточные операции не выполняются до вызова терминальной.  
**Как это работает внутри:** каждый элемент «пролетает» через всю цепочку за один проход (vertical evaluation), а не горизонтально.

```java
// filter и map не выполняются, пока не вызван toList()
list.stream()
    .filter(s -> s.length() > 3)
    .map(String::toUpperCase)
    .toList(); // 🔥 здесь всё запускается
```

### 3. Одноразовость
После вызова терминальной операции поток закрыт. Повторное использование выбрасывает `IllegalStateException`.

```java
Stream<String> s = list.stream();
s.forEach(System.out::println);
s.forEach(System.out::println); // ❌ IllegalStateException
```

### 4. Декларативный стиль
Вы описываете **что** сделать, а не **как**. Для параллельной обработки достаточно заменить `stream()` на `parallelStream()` — при условии, что операции ассоциативны и не имеют состояния.

---

## 📊 Сравнение: Stream vs цикл (for-loop)

| Характеристика | Stream API | Обычный цикл |
|----------------|------------|--------------|
| Читаемость | Высокая (цепочки) | Средняя |
| Ленивость | ✅ | ❌ (всё выполняется сразу) |
| Параллелизм | `parallelStream()` | `ForkJoinPool` вручную |
| Переиспользование | ❌ (одноразовый) | ✅ |
| Отладка | Сложнее (нет промежуточных переменных) | Легче |
| Overhead | Выше (создание объектов операций) | Минимальный |

## 🧠 Пример конвейера с объяснением

```java
List<String> result = list.stream()           // источник
    .filter(s -> s.length() > 3)              // промежуточная (ленивая)
    .map(String::toUpperCase)                 // промежуточная (ленивая)
    .sorted()                                 // промежуточная (ленивая, stateful)
    .limit(10)                                // промежуточная (ленивая, short-circuit)
    .collect(Collectors.toList());            // терминальная (запускает всё)
```

**Что происходит под капотом:**
1. Создаётся `ReferencePipeline` (внутренний класс `Stream`)
2. Каждая промежуточная операция оборачивает предыдущий поток в новый
3. При вызове `collect()` запускается `Spliterator` + `PipelineHelper`
4. Элементы проталкиваются вертикально через все `Sink`-цепочки

---

## ⚠️ Важные контракты и подводные камни

* **Non-interference:** нельзя модифицировать источник во время обработки (кроме `ConcurrentHashMap`, если используется параллельный поток)
* **Stateless vs Stateful:** `filter()`, `map()` — stateless (безопасны для параллелизма). `sorted()`, `distinct()` — stateful (хранят всё в памяти)
* **Short-circuiting:** `limit()`, `anyMatch()` — могут прервать обработку досрочно
* **Порядок выполнения:** для последовательных потоков сохраняется порядок встречи (encounter order), для параллельных — зависит

## 🔗 Связи с другими темами

* `Collection` → `.stream()` (Java 8)
* `Arrays` → утилитный класс для массивов
* `ForkJoinPool` — общий пул для `parallelStream()`
* `Spliterator` — базовый интерфейс для обхода данных в Stream
* `Collectors` — терминальные операции, возвращающие коллекции/строки/мапы

## 💡 Пример с реальной задачей

```java
// Найти топ-3 самых длинных слова без дубликатов
String text = "a aa aaa aaaa aa aaa aaaa aaaaa";
List<String> top = Arrays.stream(text.split(" "))
    .distinct()
    .sorted((a, b) -> Integer.compare(b.length(), a.length()))
    .limit(3)
    .toList();  // Java 16+: [aaaaa, aaaa, aaa]
```

### Шпаргалка

* **Stream — это одноразовый конвейер:** источник → ленивые операции → терминал
* **Промежуточные операции не работают без терминальной:** ленивость экономит ресурсы
* **Не меняет источник данных:** работает с копией или представлением
* **Замена stream() → parallelStream() = параллельная обработка** (если операции stateless)
* **Ошибка повторного использования:** `IllegalStateException` после терминала
