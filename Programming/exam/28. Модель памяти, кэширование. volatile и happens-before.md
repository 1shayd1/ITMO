# 28. Модель памяти, кэширование. `volatile` и `happens-before`

## 💡 Общая идея

В Java многопоточность зависит не только от потоков, но и от **Java Memory Model (JMM)** — модели памяти.

Она определяет:

- когда изменения одного потока видны другому
- как работают кэши процессора
- как JVM может переупорядочивать инструкции

---

# 🔴 Проблема видимости

Каждый поток может работать не напрямую с общей памятью, а с **локальным кэшем процессора**.

---

## ❗ Что может происходить

```text
Поток A изменил переменную
→ Поток B продолжает видеть старое значение
```

---

## Причина

- кэш CPU
- оптимизация JVM
- переупорядочивание инструкций

---

# 🟢 Модификатор volatile

## Что это

`volatile` гарантирует **видимость переменной между потоками**.

---

## Пример

```java
volatile boolean running = true;
```

---

## Что гарантирует volatile

✔ всегда читается актуальное значение  
✔ запись сразу видна другим потокам  
✔ запрещает часть переупорядочивания  

---

## Что НЕ гарантирует

❌ атомарность операций  

---

## Пример проблемы

```java
volatile int count = 0;

count++; // НЕ атомарно
```

---

# 🧠 Почему count++ не безопасен

```text
1. прочитать count
2. увеличить
3. записать обратно
```

→ между шагами может вмешаться другой поток

---

# 🔵 happens-before

## Что это

`happens-before` — правило, определяющее **порядок видимости операций между потоками**.

---

## Основная идея

Если действие A happens-before B:

```text
→ результат A гарантированно виден в B
```

---

# ⚙ Основные правила happens-before

## 1. volatile

```java
write volatile → happens-before → read volatile
```

---

## 2. synchronized

```java
unlock(monitor) → happens-before → lock(monitor)
```

---

## 3. Thread.start()

```java
перед start() → happens-before → выполнение нового потока
```

---

## 4. Thread.join()

```java
завершение потока → happens-before → продолжение после join()
```

---

# 🔗 Пример volatile

```java
class Flag {
    volatile boolean stop = false;
}
```

### Поток 1:

```java
stop = true;
```

### Поток 2:

```java
while (!stop) {
    // ждёт
}
```

✔ изменение гарантированно увидится

---

# ⚠ Важные особенности

## 1. volatile ≠ synchronized

| volatile | synchronized |
|----------|-------------|
| только видимость | видимость + атомарность |
| нет блокировки | есть блокировка |

---

## 2. кэширование CPU

```text
каждый поток может видеть разные значения
```

---

## 3. переупорядочивание

```text
JVM может менять порядок инструкций для оптимизации
```

---

# 💥 Частые ошибки

### ❌ считать volatile потокобезопасным

→ это не так

---

### ❌ использовать volatile для счетчиков

→ race condition

---

### ❌ игнорировать happens-before

→ неожиданные баги

---

# 🧠 Шпаргалка

- JMM управляет видимостью между потоками
- проблема: кэш CPU + оптимизация JVM
- `volatile` гарантирует видимость
- `volatile` НЕ гарантирует атомарность
- `happens-before` описывает порядок видимости
- `synchronized` также создает happens-before
