# 39. Паттерны проектирования. Порождающие паттерны. Примеры

**Порождающие паттерны** (creational patterns) — это паттерны проектирования, которые абстрагируют процесс создания объектов, делая систему независимой от способа их порождения, композиции и представления.

---

## 📋 Классификация порождающих паттернов (GoF)

| Паттерн              | Суть                                                                | Когда использовать                                      |
| -------------------- | ------------------------------------------------------------------- | ------------------------------------------------------- |
| **Singleton**        | Гарантирует единственный экземпляр класса                           | Глобальная точка доступа (логи, конфиги, пулы)          |
| **Factory Method**   | Создание объекта через метод, который переопределяется в подклассах | Тип объекта заранее неизвестен, но иерархия фиксирована |
| **Abstract Factory** | Создание семейств связанных объектов                                | Нужны целые наборы объектов (UI для разных ОС)          |
| **Builder**          | Пошаговое создание сложного объекта                                 | Много параметров, некоторые опциональны                 |
| **Prototype**        | Клонирование объектов                                               | Создание дороже копирования                             |

## 1. Singleton (Одиночка)

Гарантирует, что у класса есть только **один экземпляр**, и предоставляет глобальную точку доступа к нему.

### Реализации в Java (от худшей к лучшей)

#### ❌ Ленивая (lazy) — не thread-safe
```java
public class SingletonLazy {
    private static SingletonLazy instance;
    private SingletonLazy() {}
    
    public static SingletonLazy getInstance() {
        if (instance == null) {     // ❌ race condition
            instance = new SingletonLazy();
        }
        return instance;
    }
}
```

#### ⚠️ Синхронизированный метод — медленно
```java
public class SingletonSync {
    private static SingletonSync instance;
    private SingletonSync() {}
    
    public static synchronized SingletonSync getInstance() {
        if (instance == null) {
            instance = new SingletonSync();
        }
        return instance;
    }
}
```

#### ✅ Double-Checked Locking (с volatile)
```java
public class SingletonDCL {
    private static volatile SingletonDCL instance;
    private SingletonDCL() {}
    
    public static SingletonDCL getInstance() {
        if (instance == null) {
            synchronized (SingletonDCL.class) {
                if (instance == null) {
                    instance = new SingletonDCL();
                }
            }
        }
        return instance;
    }
}
```

#### ✅✅ Лучший способ — Enum (Joshua Bloch)
```java
public enum SingletonEnum {
    INSTANCE;
    
    public void doSomething() {
        System.out.println("Singleton action");
    }
}
// Использование: SingletonEnum.INSTANCE.doSomething()
```

#### ✅✅✅ Статическая инициализация (eager loading)
```java
public class SingletonEager {
    private static final SingletonEager INSTANCE = new SingletonEager();
    private SingletonEager() {}
    public static SingletonEager getInstance() { return INSTANCE; }
}
```

### Подкапотные детали Singleton

- **Проблема сериализации:** обычный синглтон при десериализации создаёт новый объект. Решение — `readResolve()`.
- **Reflection:** может вызвать приватный конструктор. Enum защищает от этого.
- **ClassLoader:** разные загрузчики = разные синглтоны.

```java
// Защита от рефлексии в конструкторе
private Singleton() {
    if (instance != null) {
        throw new IllegalStateException("Already instantiated");
    }
}

// Защита от сериализации
protected Object readResolve() {
    return getInstance();
}
```

## 2. Factory Method (Фабричный метод)

Определяет интерфейс для создания объекта, но позволяет подклассам решать, какой именно класс инстанцировать.

### Структура и пример

```java
// Product
interface Transport {
    void deliver();
}

// Concrete products
class Truck implements Transport {
    public void deliver() { System.out.println("Deliver by truck"); }
}

class Ship implements Transport {
    public void deliver() { System.out.println("Deliver by ship"); }
}

// Creator (фабричный метод)
abstract class Logistics {
    abstract Transport createTransport(); // factory method
    
    public void planDelivery() {
        Transport t = createTransport();
        t.deliver();
    }
}

// Concrete creators
class RoadLogistics extends Logistics {
    Transport createTransport() { return new Truck(); }
}

class SeaLogistics extends Logistics {
    Transport createTransport() { return new Ship(); }
}
```

### Когда полезен

- Заранее неизвестно, объекты каких классов создавать
- Подклассы должны расширять/изменять тип создаваемых объектов
- Код, использующий фабрику, работает с абстракциями

### Complexity & подкапот

- **Time:** O(1) — просто создание объекта
- **Memory:** Минимальный overhead (одна дополнительная иерархия)

## 3. Abstract Factory (Абстрактная фабрика)

Создаёт **семейства связанных** или взаимозависимых объектов, не привязываясь к их конкретным классам.

### Пример: кроссплатформенный UI

```java
// Abstract products
interface Button { void render(); }
interface Checkbox { void render(); }

// Concrete products for Windows
class WinButton implements Button { public void render() { System.out.println("Windows button"); } }
class WinCheckbox implements Checkbox { public void render() { System.out.println("Windows checkbox"); } }

// Concrete products for Mac
class MacButton implements Button { public void render() { System.out.println("Mac button"); } }
class MacCheckbox implements Checkbox { public void render() { System.out.println("Mac checkbox"); } }

// Abstract factory
interface GUIFactory {
    Button createButton();
    Checkbox createCheckbox();
}

// Concrete factories
class WinFactory implements GUIFactory {
    public Button createButton() { return new WinButton(); }
    public Checkbox createCheckbox() { return new WinCheckbox(); }
}

class MacFactory implements GUIFactory {
    public Button createButton() { return new MacButton(); }
    public Checkbox createCheckbox() { return new MacCheckbox(); }
}
```

### Использование

```java
class Application {
    private final Button button;
    private final Checkbox checkbox;
    
    Application(GUIFactory factory) {
        button = factory.createButton();
        checkbox = factory.createCheckbox();
    }
    
    void render() {
        button.render();
        checkbox.render();
    }
}

// Клиентский код
Application app = new Application(new MacFactory());
app.render(); // Mac button, Mac checkbox
```

### Abstract Factory vs Factory Method

| Характеристика | Factory Method | Abstract Factory |
|----------------|----------------|------------------|
| Уровень абстракции | Один продукт | Семейство продуктов |
| Реализация | Наследование | Композиция |
| Количество методов | Один | Несколько (по числу продуктов) |
| Изменение семейства | Нет (новый подкласс) | Да (новая конкретная фабрика) |

## 4. Builder (Строитель)

Разделяет конструирование сложного объекта от его представления, позволяя одному и тому же процессу создавать разные представления.

### Классический Builder (GoF)

```java
// Product
class Pizza {
    private String dough;
    private String sauce;
    private String topping;
    
    private Pizza() {} // private constructor
    
    // Getters...
    
    // Builder
    static class Builder {
        private Pizza pizza = new Pizza();
        
        Builder dough(String dough) { pizza.dough = dough; return this; }
        Builder sauce(String sauce) { pizza.sauce = sauce; return this; }
        Builder topping(String topping) { pizza.topping = topping; return this; }
        
        Pizza build() {
            if (pizza.dough == null) throw new IllegalStateException("Dough required");
            return pizza;
        }
    }
}

// Использование
Pizza pizza = new Pizza.Builder()
    .dough("thin")
    .sauce("tomato")
    .topping("cheese")
    .build();
```

### Современный Builder (рекорды + строитель)

```java
public record User(
    Long id,
    String email,
    String firstName,
    String lastName,
    Integer age,
    String phone
) {
    public static Builder builder() {
        return new Builder();
    }
    
    public static class Builder {
        private Long id;
        private String email;
        private String firstName;
        private String lastName;
        private Integer age;
        private String phone;
        
        public Builder id(Long id) { this.id = id; return this; }
        public Builder email(String email) { this.email = email; return this; }
        public Builder firstName(String firstName) { this.firstName = firstName; return this; }
        public Builder lastName(String lastName) { this.lastName = lastName; return this; }
        public Builder age(Integer age) { this.age = age; return this; }
        public Builder phone(String phone) { this.phone = phone; return this; }
        
        public User build() {
            return new User(id, email, firstName, lastName, age, phone);
        }
    }
}
```

### Когда использовать Builder

- Конструктор имеет 4+ параметров (особенно однотипных)
- Многие параметры опциональны
- Объект неизменяемый (immutable)
- Создание сложное (валидация, зависимости между параметрами)

### Complexity

- **Time:** O(k) где k — число установленных полей
- **Memory:** Временный объект билдера (+ overhead)

## 5. Prototype (Прототип)

Создаёт новые объекты **копированием** существующего (прототипа), вместо вызова конструктора.

### Реализация с Cloneable (не рекомендуется, но классическая)

```java
class Shape implements Cloneable {
    private String type;
    private int x, y;
    
    public Shape clone() {
        try {
            return (Shape) super.clone(); // shallow copy
        } catch (CloneNotSupportedException e) {
            return null;
        }
    }
}
```

### Современная реализация (копирующий конструктор)

```java
class Document {
    private String title;
    private List<String> paragraphs;
    private Map<String, String> metadata;
    
    // Обычный конструктор
    public Document(String title, List<String> paragraphs, Map<String, String> metadata) {
        this.title = title;
        this.paragraphs = new ArrayList<>(paragraphs);
        this.metadata = new HashMap<>(metadata);
    }
    
    // ✅ Копирующий конструктор (deep copy)
    public Document(Document other) {
        this.title = other.title;
        this.paragraphs = new ArrayList<>(other.paragraphs);
        this.metadata = new HashMap<>(other.metadata);
    }
    
    // Фабричный метод копирования
    public Document copy() {
        return new Document(this);
    }
}

// Использование
Document original = new Document("Report", List.of("p1", "p2"), Map.of("author", "John"));
Document copy = original.copy();
```

### Registry прототипов

```java
class PrototypeRegistry {
    private Map<String, Document> prototypes = new HashMap<>();
    
    public void register(String key, Document prototype) {
        prototypes.put(key, prototype);
    }
    
    public Document create(String key) {
        return prototypes.get(key).copy();
    }
}

// Использование
registry.register("invoice", new InvoiceTemplate());
Document invoice = registry.create("invoice");
```

### Когда использовать Prototype

- Создание объекта дороже копирования (например, после сложного запроса)
- Объекты различаются лишь несколькими параметрами
- Нужно избежать иерархии фабрик

### Complexity

- **Shallow copy:** O(n) где n — число примитивных полей
- **Deep copy:** O(m) где m — суммарный размер графа объектов
- **Memory:** Копия занимает столько же, сколько оригинал

## 📊 Сравнительная таблица всех порождающих паттернов

| Паттерн | Проблема | Решение | Complexity | Популярность |
|---------|----------|---------|------------|--------------|
| **Singleton** | Один экземпляр | Приватный конструктор + статическое поле | O(1) | ⭐⭐⭐⭐⭐ |
| **Factory Method** | Тип объекта неизвестен | Виртуальный конструктор в подклассах | O(1) | ⭐⭐⭐⭐⭐ |
| **Abstract Factory** | Семейства объектов | Интерфейс фабрики с несколькими методами | O(1) на продукт | ⭐⭐⭐⭐ |
| **Builder** | Много параметров | Пошаговое построение с fluent API | O(k) | ⭐⭐⭐⭐⭐ |
| **Prototype** | Дорогое создание | Клонирование (shallow/deep) | O(n) на копию | ⭐⭐⭐ |

## 🧠 Реальный пример — комбинация Builder + Abstract Factory

```java
// Конфигурация приложения с использованием нескольких паттернов
public class AppConfig {
    private final DatabaseConfig dbConfig;
    private final CacheConfig cacheConfig;
    private final LogConfig logConfig;
    
    private AppConfig(Builder builder) {
        this.dbConfig = builder.dbConfig;
        this.cacheConfig = builder.cacheConfig;
        this.logConfig = builder.logConfig;
    }
    
    public static class Builder {
        private DatabaseConfig dbConfig = DatabaseConfig.defaultConfig();
        private CacheConfig cacheConfig = CacheConfig.defaultConfig();
        private LogConfig logConfig = LogConfig.defaultConfig();
        
        public Builder database(DatabaseConfig config) { this.dbConfig = config; return this; }
        public Builder cache(CacheConfig config) { this.cacheConfig = config; return this; }
        public Builder logging(LogConfig config) { this.logConfig = config; return this; }
        
        public AppConfig build() {
            return new AppConfig(this);
        }
    }
}

// Abstract Factory для создания конфигураций под окружение
interface ConfigFactory {
    DatabaseConfig createDatabaseConfig();
    CacheConfig createCacheConfig();
}

class DevConfigFactory implements ConfigFactory {
    public DatabaseConfig createDatabaseConfig() { return new DatabaseConfig("localhost", 5432); }
    public CacheConfig createCacheConfig() { return new CacheConfig(100, "memory"); }
}

class ProdConfigFactory implements ConfigFactory {
    public DatabaseConfig createDatabaseConfig() { return new DatabaseConfig("prod-db", 5432); }
    public CacheConfig createCacheConfig() { return new CacheConfig(10000, "redis"); }
}

// Использование всех трёх паттернов
ConfigFactory factory = isDev() ? new DevConfigFactory() : new ProdConfigFactory();
AppConfig config = new AppConfig.Builder()
    .database(factory.createDatabaseConfig())
    .cache(factory.createCacheConfig())
    .build();
```

## ⚠️ Антипаттерны и частые ошибки

### Singleton
- ❌ Использовать для тяжёлых объектов с ранней инициализацией (замедляет старт)
- ❌ Синглтон с состоянием в многопоточной среде
- ✅ Использовать DI-контейнер (Spring) вместо Singleton

### Builder
- ❌ Создавать билдер для 1-2 параметров (избыточно)
- ❌ Не делать валидацию в `build()`

### Factory Method
- ❌ Фабричный метод, который всегда возвращает один тип (просто `new`)
- ✅ Переопределяемый метод с реальной логикой выбора

### Prototype
- ❌ Использовать `Cloneable` без deep copy для mutable полей
- ✅ Лучше копирующий конструктор или фабрика копирования

## 🔗 Связи с другими темами Java

- **Singleton** + `Enum` — защита от рефлексии/сериализации
- **Builder** + `Record` — идеальное сочетание для DTO
- **Factory Method** + `Stream` (например, `Collector.of()`)
- **Prototype** + `Object.clone()` (но с осторожностью)
- В Spring: бины по умолчанию — **Singletons**, `@Scope("prototype")`, `FactoryBean`

---

### Шпаргалка

* **Singleton:** один экземпляр на JVM — `enum` или `static final`
* **Factory Method:** создание через метод в иерархии — один продукт
* **Abstract Factory:** семейство продуктов — несколько фабричных методов
* **Builder:** пошаговое создание с fluent API — для 4+ параметров
* **Prototype:** клонирование через копирующий конструктор — дорогое создание
* **Все порождающие паттерны скрывают логику создания объектов** — клиент работает с интерфейсами
* **Выбор паттерна = компромисс между гибкостью и сложностью**
