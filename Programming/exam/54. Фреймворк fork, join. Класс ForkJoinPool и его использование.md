# 54. Фреймворк fork/join. Класс ForkJoinPool и его использование

**Fork/Join** (Java 7) — фреймворк для распараллеливания задач по принципу **«разделяй и властвуй»**: задача рекурсивно делится на подзадачи (`fork`), которые выполняются параллельно, а затем результаты объединяются (`join`).

## Идея

* **fork** — отделить подзадачу и поставить её в очередь на асинхронное выполнение;
* **join** — дождаться завершения подзадачи и получить её результат;
* деление продолжается до **порога** (threshold), ниже которого задача решается последовательно — это критично, т.к. накладные расходы на создание задач могут «съесть» выигрыш от параллелизма.

## Work-stealing — «подкапотный» механизм

Ключевое отличие `ForkJoinPool` от обычного `ThreadPoolExecutor` — алгоритм **похищения работы**:

* у каждого рабочего потока есть **собственная двусторонняя очередь** (deque) задач;
* **владелец** берёт задачи со своего конца дека в порядке **LIFO** (свежие подзадачи — горячие в кэше);
* **простаивающий поток-«вор»** похищает задачи с противоположного конца чужого дека в порядке **FIFO** (старые задачи — самые крупные, их выгоднее красть);
* за счёт этого нагрузка балансируется автоматически, а конкуренция за очереди минимальна.

## `ForkJoinPool`

* `ForkJoinPool.commonPool()` — общий статический пул; по умолчанию имеет **`Runtime.getRuntime().availableProcessors() − 1`** рабочих потоков. Именно его используют **параллельные стримы** и `CompletableFuture` без явного executor'а.
* `new ForkJoinPool(parallelism)` — отдельный пул с заданным параллелизмом.
* Запуск: `pool.invoke(task)` — синхронно с результатом; `pool.submit(task)` — асинхронно (`Future`); `pool.execute(task)` — «выстрелил и забыл».

## `RecursiveTask` и `RecursiveAction`

Обе — наследники `ForkJoinTask<V>`, переопределяют метод `compute()`:

| Класс | Результат | Аналог |
|---|---|---|
| `RecursiveTask<V>` | возвращает `V` | `Callable` |
| `RecursiveAction` | не возвращает (`void`) | `Runnable` |

## Пример: сумма массива

```java
class SumTask extends RecursiveTask<Long> {
    private static final int THRESHOLD = 10_000;
    private final long[] arr;
    private final int from, to;

    SumTask(long[] arr, int from, int to) {
        this.arr = arr; this.from = from; this.to = to;
    }

    @Override
    protected Long compute() {
        if (to - from <= THRESHOLD) {              // база рекурсии
            long sum = 0;
            for (int i = from; i < to; i++) sum += arr[i];
            return sum;
        }
        int mid = (from + to) >>> 1;
        SumTask left  = new SumTask(arr, from, mid);
        SumTask right = new SumTask(arr, mid, to);
        right.fork();                  // правую — асинхронно
        long l = left.compute();       // левую — в текущем потоке (экономим fork)
        return l + right.join();       // дождаться правую
    }
}

long total = ForkJoinPool.commonPool()
        .invoke(new SumTask(data, 0, data.length));
```

**Идиома:** `fork()` одну половину, `compute()` другую в текущем потоке, затем `join()`. Вызов `fork()` для обеих половин с последующими двумя `join()` создаёт лишнюю задачу. Для нескольких задач удобен `invokeAll(t1, t2, ...)`.

## Где полезен

* обработка больших массивов и коллекций (сортировка `Arrays.parallelSort()` использует fork/join);
* рекурсивные вычисления (обход деревьев, файловой системы);
* **CPU-bound** задачи, делящиеся на независимые части. Для блокирующего I/O фреймворк подходит плохо: поток, заблокированный на `join()` чужой медленной задачи, может при этом выполнять другие задачи из очереди (work-stealing), но блокировка на внешнем I/O «выключает» поток из пула.

### Шпаргалка

* Fork/Join = «разделяй и властвуй» + **work-stealing**: свой дек LIFO, кража чужих задач FIFO.
* `RecursiveTask<V>` — с результатом, `RecursiveAction` — без.
* `commonPool()` имеет `CPU − 1` потоков и используется параллельными стримами.
* Идиома: `right.fork(); left.compute(); right.join();` + порог последовательного выполнения.
* Подходит для CPU-bound рекурсивно делимых задач, а не для блокирующего I/O.
