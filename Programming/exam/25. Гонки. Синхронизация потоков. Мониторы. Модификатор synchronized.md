# 25. Гонки. Синхронизация потоков. Мониторы. Модификатор `synchronized`

## 💡 Общая идея

При многопоточности несколько потоков могут одновременно работать с одними и теми же данными. Это приводит к проблеме **гонки данных (race condition)**.

---

# 🔴 Гонки (Race Condition)

## Что это

**Гонка данных** — это ситуация, когда результат зависит от порядка выполнения потоков.

---

## Пример проблемы

```java
class Counter {
    private int value = 0;

    void inc() {
        value++;
    }
}
```

---

## Почему это ошибка

Операция:

```text
value++
```

не атомарна и включает:

1. чтение value
2. увеличение
3. запись обратно

---

## Пример гонки

```text
Поток A: читает value = 5
Поток B: читает value = 5
Поток A: записывает 6
Поток B: записывает 6 (потеря инкремента)
```

---

# 🟢 Синхронизация потоков

## Что это

Синхронизация нужна, чтобы **разрешить доступ к ресурсу только одному потоку одновременно**.

---

# 🔵 synchronized

## Что делает

`sychronized` обеспечивает **взаимное исключение (mutual exclusion)**.

---

## Синхронизированный метод

```java
synchronized void inc() {
    value++;
}
```

---

## Синхронизированный блок

```java
void inc() {
    synchronized (this) {
        value++;
    }
}
```

---

## Разница

| Вариант | Где блокировка |
|--------|---------------|
| synchronized method | весь метод |
| synchronized block | часть кода |

---

# 🧠 Монитор (Monitor)

## Что это

Каждый объект в Java имеет **монитор** — встроенный механизм блокировки.

---

## Как работает монитор

```text
поток → захватывает монитор объекта
      → выполняет synchronized код
      → освобождает монитор
```

---

## Пример

```java
synchronized (obj) {
    // только один поток может здесь быть
}
```

---

# ⚙ Внутренний механизм

## Вход в synchronized

```text
thread → lock(monitor)
```

## Выход

```text
thread → unlock(monitor)
```

---

# ⚠ Проблемы синхронизации

## 1. Снижение производительности

```text
много потоков → ожидание блокировки
```

---

## 2. Deadlock (взаимная блокировка)

```text
Thread A ждёт B
Thread B ждёт A
```

---

## 3. Contention

```text
постоянная борьба за один монитор
```

---

# 💥 Частые ошибки

### ❌ отсутствие synchronized

→ гонки данных

---

### ❌ слишком большая синхронизация

→ падение производительности

---

### ❌ разные мониторы для одного ресурса

→ защита не работает

---

# 🔗 Пример правильного решения

```java
class Counter {
    private int value = 0;

    synchronized void inc() {
        value++;
    }

    int get() {
        return value;
    }
}
```

---

# 🧠 Шпаргалка

* гонка данных = одновременный доступ к общим данным
* `value++` не атомарна
* `synchronized` защищает критическую секцию
* монитор = механизм блокировки объекта
* synchronized может быть методом или блоком
* неправильная синхронизация → deadlock / падение производительности
